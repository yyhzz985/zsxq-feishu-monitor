# 锤子便签本地渲染部署手册

## 目标状态

- 官方 `zhaoolee/notes` 容器只监听 `127.0.0.1:18080`。
- ZSXQ 四个实例与 `qq-bridge-wu2198` 共用 `/run/lock/hermes-notes-renderer.lock`。
- 正常请求由官方 Playwright 渲染；失败时立即使用本地 Pillow 渲染同一份文字和图片。
- 不再访问 `notes.fangyuanxiaozhan.com`。

## 上线前检查

在本地仓库运行：

```powershell
python deploy\validate_notes_renderer.py
python -m unittest discover -s tests -v
python qq-feishu-bridge\tests\test_bridge_wu2198_renderer.py -v
python -m py_compile src\note_renderer.py src\local_notes_fallback.py src\zsxq_monitor.py qq-feishu-bridge\bridge_wu2198.py
```

所有命令成功后才允许上传。不要上传 `.env`、数据库、token 或日志。

## 生产部署

以下操作会拉取并启动容器、覆盖生产脚本，属于生产发布。必须取得明确批准后执行：

```bash
cd /path/to/Hermes
sudo bash deploy/install_notes_renderer.sh
```

脚本会：

1. 创建 `/opt/notes-renderer/storage/images` 和字体目录。
2. 启动锁定 digest 的 `hermes-notes` 容器并检查 `/api/health`。
3. 从官方容器提取 OPPOSans 字体供 Pillow 降级使用。
4. 同步渲染模块、四个 ZSXQ 入口和 wu2198 入口。
5. 只做语法检查，不重启任何生产 systemd 服务。

## 灰度验证

取得第二次明确批准后，先重启测试链路并发送一条文字加图片的测试内容。确认样式、图片、告警和资源占用后，再重启正式服务：

```bash
systemctl restart zsxq-poll.service
systemctl restart zsxq-poll-honghao.service
systemctl restart zsxq-poll-kkkkk.service
systemctl restart zsxq-poll-ggggg.service
systemctl restart qq-bridge-wu2198.service
```

验收命令：

```bash
curl -fsS http://127.0.0.1:18080/api/health
docker stats hermes-notes --no-stream
systemctl is-active zsxq-poll.timer zsxq-poll-honghao.timer zsxq-poll-kkkkk.timer zsxq-poll-ggggg.timer
systemctl is-active qq-bridge-wu2198.service
journalctl -u zsxq-poll.service -n 80 --no-pager
journalctl -u qq-bridge-wu2198.service -n 80 --no-pager
```

通过标准：容器健康；渲染峰值受资源限制；宿主机无 OOM；文字和图片位置正确；强制停止容器后日志出现 `RENDERER_FALLBACK`，飞书仍收到包含图片的便签。
