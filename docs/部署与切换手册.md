## 2. 飞书 OpenAPI 模式

`.env` 需要有这些配置：

```text
ZSXQ_ACCESS_TOKEN=不要写进文档
FEISHU_SEND_MODE=openapi
FEISHU_APP_ID=不要写进文档
FEISHU_APP_SECRET=不要写进文档
FEISHU_CHAT_ID=oc_xxx
FEISHU_CONTENT_CHAT_IDS=oc_formal_xxx  # 可选，额外内容群
FEISHU_ALERT_CHAT_ID=oc_test_xxx       # 可选，测试/告警群
```

发送规则：星球图片和附件会发到 `FEISHU_CHAT_ID`、`FEISHU_CONTENT_CHAT_IDS`、`FEISHU_ALERT_CHAT_ID`；告警和健康日报只发到 `FEISHU_ALERT_CHAT_ID`。如果没配置 `FEISHU_ALERT_CHAT_ID`，告警会发到 `FEISHU_CHAT_ID`。

OpenAPI 已验证：

```text
文本发送：成功
图片发送：成功
文件发送：成功
```

如果需要临时切回旧 lark-cli：

```text
FEISHU_SEND_MODE=cli
```

切换后必须运行：

```powershell
python C:\Users\1\.hermes\scripts\zsxq_monitor.py --check
```

## 6. 当前已知风险

ZSXQ 可能偶发返回：

```text
code=1059，不支持非官方工具访问
```

当前处理：

```text
写日志。
当天同类告警只发一次。
不推进 last_seen_time。
下轮继续重试。
```

这属于外部接口风险，代码只能做到发现、告警和不漏游标。

## 7. 附件长文件名故障处理

已修复的故障模式：

```text
[Errno 36] File name too long
```

根因是 Linux 文件名限制按 UTF-8 字节计算，不能只按字符数截断。当前代码已用 `FILENAME_MAX_BYTES=180` 限制附件临时名和归档名，并保留扩展名。

如果生产中仍有单条附件卡在 `permanent_fail`，处理顺序：

```text
1. 先查 journalctl 和 SQLite，确认具体 file_key。
2. 只重置目标 file_key，不批量改 files 表。
3. 用 systemd 启动对应 service，必须使用生产 EnvironmentFile。
4. 补发后确认 DB status=sent、last_error=null、归档文件存在。
```

2026-06-14 已补发案例：

```text
topic_id=45544551455244828
file_key=45544551455244828:181244582422282
结果：FILE_SENT，归档文件大小 6117989 bytes
```
