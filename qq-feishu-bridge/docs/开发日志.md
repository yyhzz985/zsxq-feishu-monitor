   - **断线重连**：指数退避（10s → 20s → 40s → ... → max 300s），断线 5 分钟告警
   - **SQLite 状态机**：消息入库去重（message_id 主键），sent/failed 状态追踪，失败重试队列

   - **多群支持**：通过 `config.json` 配置目标群列表，未配置则处理所有群
   - **定期失败重试**：每 5 分钟扫描 failed 队列，retry_count ≤ 3 的消息自动重发
   - **告警系统**：断线告警、飞书发送失败日志

4. **Docker 部署文件**
   - `deploy/Dockerfile`：Python 3.11-slim + websockets

### 技术决策

| 决策 | 理由 |
|------|------|
| 飞书用 OpenAPI 而非 Webhook | Webhook 不支持图片/文件上传，且格式受限 |
| 异步 WebSocket (asyncio) | 不能阻塞消息接收，需并发处理 |
| Docker 内 Python 3.11 | 阿里云服务器 Python 3.6 不支持 websockets 12.x，Docker 解决依赖问题 |
| 复用 ZSXQ 飞书模块 | 避免重复造轮子，两个项目共享同一套飞书 Bot 认证 |
| 单文件脚本 | KISS 原则，先跑通再拆模块 |
| SQLite 本地持久化 | 轻量、零配置、与 ZSXQ Monitor 同模式 |

### 踩坑记录

- **飞书 Webhook 限制**：Claude 方案用 Webhook，实测只能发 text/post 类型，图片需要先上传再发 image_key，Webhook 不支持上传。改用 OpenAPI（Bot API）。
- **OneBot 消息段格式**：消息内容是 `[{"type":"text","data":{"text":"xxx"}}, ...]` 的数组，不是纯字符串。Claude 方案的解析代码验证可用，直接复用。
- **tenant_access_token 缓存**：复用 ZSXQ Monitor 的缓存逻辑（提前 5 分钟刷新），避免每次发消息都调 token API。
- **飞书多 Bot 优先级**：环境变量 `ZSXQ_FEISHU_APP_ID` 优先于 `FEISHU_APP_ID`，与 ZSXQ Monitor 保持一致，方便共享"汤圆"同步专用 Bot。

### 架构确定

```
QQ群消息 → NapCat Docker (OneBot v11)
    ├── WebSocket 实时推送 → Bridge 容器
    │   ├── 文本 → 飞书文本消息
    │   ├── 图片 URL → 下载 → 飞书图片上传
    │   └── 文件 URL → 下载 → 飞书文件上传
    └── HTTP API → 历史消息拉取 (get_group_msg_history)
```

### 关键发现

- **图片消息无 file_id**：OneBot 事件中图片只有 `file`(文件名) + `url`，没有 `file_id`
- **文件消息有 file_id**：格式如 `/7c1edf90-5af7-11f1-8bd5-5254001daf95`
- **QQ CDN URL 可直链下载**：multimedia.nt.qq.com.cn 的 URL 带 token 参数，有效期较长
- **NapCat 端口 3000 被宝塔占用**：改用 3099 映射
- **Alibaba Cloud Linux 3 防火墙**：需手动 `firewall-cmd --add-port=6099/tcp`
