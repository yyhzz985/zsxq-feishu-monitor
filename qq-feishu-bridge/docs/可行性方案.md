# QQ群消息 → 飞书群 转发系统：可行性落地方案

> 创建日期：2026-05-30
> 状态：方案阶段，未编码

---

## 一、需求背景

用户需要监控别人创建的 QQ 群（用户是群成员，非群主/管理员），将群内消息（文本、图片、文件）转发到飞书群。

与现有 ZSXQ Monitor 的关系：
- ZSXQ Monitor：知识星球 → 便签图片 + 水印 → 飞书（**生产级，已在阿里云运行**）
- QQ Bridge：QQ 群 → 格式化消息 → 飞书（**本项目，方案阶段**）

---

## 二、技术路径对比

### 2.1 腾讯 QQ 无公开 API

QQ 不提供群消息读取 API。所有可行方案都是"非官方路径"：

| 方案 | 原理 | 服务器部署 | 风控风险 | 维护成本 |
|------|------|-----------|---------|---------|
| **NapCat + WebSocket** ✅ | 注入 QQ NT 进程，拦截消息事件 | Docker | 低~中 | 中（QQ更新需适配） |
| Mirai + 签名服务 | 协议层模拟 QQ 登录 | ✅ | 高 | 高（签名不稳定） |
| OCR 截图抓取 | 定时截图 + 文字识别 | 需要 GUI | 极低 | 高（漏消息严重） |
| qq-cli 数据库直读 | 读 NTQQ SQLCipher 数据库 | ❌ 仅 macOS | 无 | 低（但平台受限） |

**结论：NapCat 是当前唯一可行路径。** 它是 QQ NT Hook 方案中最成熟、社区最活跃的项目。

### 2.2 Claude 方案的评估

Claude 生成的方案（`qq-feishu-forwarder.zip`）核心链路：

```
NapCat Docker → WebSocket → Python转发脚本 → 飞书 Webhook
```

**可用部分**：
- NapCat Docker 部署方案 ✅
- OneBot 消息段解析逻辑 ✅
- WebSocket 断线重连 ✅

**不可用部分**：
- 飞书 Webhook 只能发文本，不能发图片/文件 ❌
- 无状态追踪，断线消息永远丢失 ❌
- 无失败重试机制 ❌

### 3.2 与 ZSXQ Monitor 的代码复用

两个项目的飞书发送逻辑完全相同，**不需要重写**：

| 模块 | ZSXQ Monitor | QQ Bridge | 复用方式 |
|------|-------------|-----------|---------|
| `feishu_send_text_openapi()` | ✅ | ✅ | 直接复制 |
| `feishu_send_image_openapi()` | ✅ | ✅ | 直接复制 |
| `feishu_send_file_openapi()` | ✅ | ✅ | 直接复制 |
| `get_feishu_tenant_token()` | ✅ | ✅ | 直接复制 |
| `feishu_upload_image_openapi()` | ✅ | ✅ | 直接复制 |
| `send_alert()` | ✅ | ✅ | 直接复制 |
| SQLite 状态机 | ✅ | ✅ | 相同 schema，独立 DB |
| 文件锁防并发 | ✅ | ✅ | 直接复制 |
| 日志模块 | ✅ | ✅ | 直接复制 |

**不共享的部分**：
- 消息获取层：ZSXQ 是 HTTP 轮询，QQ 是 WebSocket 事件推送
- 便签导出 + 水印：QQ Bridge 不需要（至少 MVP 阶段不用）

### 3.3 SQLite 状态机

```sql
-- qq_messages 表：每条 QQ 消息独立追踪
CREATE TABLE qq_messages (
    message_id  TEXT PRIMARY KEY,    -- OneBot message_id (全局唯一)
    group_id    INTEGER NOT NULL,    -- QQ 群号
    sender_uin  INTEGER,            -- 发送者 QQ 号
    sender_name TEXT,               -- 群名片/昵称
    msg_type    TEXT,               -- text / image / file / ...
    content     TEXT,               -- 消息内容 (文本) 或 JSON (多段)
    raw_json    TEXT,               -- 原始 OneBot 事件 JSON
    status      TEXT DEFAULT 'received',  -- received / sent / failed
    feishu_msg_id TEXT,             -- 飞书消息 ID
    retry_count INTEGER DEFAULT 0,
    last_error  TEXT,
    created_at  TEXT,
    updated_at  TEXT
);

-- qq_media 表：图片/文件媒体（独立追踪，下载可能失败）
CREATE TABLE qq_media (
    media_key   TEXT PRIMARY KEY,   -- {message_id}:{file_unique}
    message_id  TEXT,               -- 关联消息
    media_type  TEXT,               -- image / file / voice
    file_name   TEXT,
    file_url    TEXT,               -- QQ 图片/文件 URL
    local_path  TEXT,               -- 下载后的本地路径
    status      TEXT DEFAULT 'pending',  -- pending / downloaded / uploaded / failed
    retry_count INTEGER DEFAULT 0,
    last_error  TEXT,
    created_at  TEXT,
    updated_at  TEXT,
    FOREIGN KEY (message_id) REFERENCES qq_messages(message_id)
);

-- meta 表：心跳、连续失败计数、断线时间
CREATE TABLE meta (
    key         TEXT PRIMARY KEY,
    value       TEXT,
    updated_at  TEXT
);
```

## 四、消息处理流程

### 4.1 正常流程

```
WebSocket 事件
    │
    ▼
[message_id 去重] → 已处理？ → 跳过
    │ 未处理
    ▼
[入库 qq_messages] → status='received'
    │
    ├── 文本消息 → 格式化 → feishu_send_text_openapi()
    │
    ├── 图片消息 → 下载 QQ 图片(需Cookie) → 上传飞书 → feishu_send_image_openapi()
    │
    ├── 文件消息 → 下载 QQ 文件(需Cookie) → 上传飞书 → feishu_send_file_openapi()
    │
    └── 语音/视频 → 标记 [语音消息] / [视频消息] → feishu_send_text_openapi()
    │
    ▼
[更新状态] → sent (成功) 或 failed (失败, 等待重试)
```

### 4.2 断线恢复流程

```
每次启动时：
1. 检查 meta.last_disconnect_time
2. 如果距上次断开超过 1 分钟 → 标记可能存在消息遗漏
3. WebSocket 重连成功后 → 正常接收实时消息
4. ⚠️ 断线期间的消息无法回溯（OneBot 不提供历史消息 API）

缓解措施：
- 每 30 秒的心跳检测，断线快速发现
- 自动重连（间隔递增：10s → 30s → 60s → ...）
- 断线超过 5 分钟 → 飞书告警
- 记录断线时间段到 meta 表
```

### 4.3 失败重试

```
qq_messages.status = 'failed' 的记录：
- WebSocket 事件驱动处理（实时）
- 定期（每 5 分钟）扫描 failed 队列
- retry_count <= 3 → 重试发送
- retry_count > 3 → 放弃，发告警
```

## 五、关键问题与解决方案

### 5.1 QQ 图片/文件下载

**问题**：OneBot 事件中图片 URL 是 QQ 内部链接（如 `http://gchat.qpic.cn/...`），需要登录态 Cookie 才能下载。

**方案**：
- NapCat 可以配置暴露 HTTP API（含 Cookie）
- 或从 NapCat Docker 容器内 `/app/.config/QQ/` 直接访问缓存文件
- **推荐**：使用 NapCat 的 HTTP API `get_image` / `get_file` 获取文件数据

### 5.2 QQ Cookie/登录态维持

**问题**：QQ 登录后 Token 可能过期；服务器 IP 可能触发风控。

**缓解**：
- 使用专用 QQ 小号（不聊天、不加好友，纯粹监控）
- NapCat 支持 ticket 刷新（QQ NT 原生机制）
- 服务器选国内节点（阿里云华东/华北）
- 首次登录建议在本地完成，再传 `napcat_data` 到服务器

### 5.3 飞书消息格式

**问题**：QQ 群消息格式与 ZSXQ 帖子格式差异大。是否需要便签图？

**MVP 方案（不生成便签图）**：
```
【QQ群·击球区交流群】
狐叔 2026-05-30 15:32
今天XX标的有什么看法？

[图片] ← 直接上传原图到飞书
```

**完全体方案（生成便签图）**：
- 可复用 ZSXQ Monitor 的便签导出 + 水印逻辑
- 但每张图都生成便签 → 飞书刷屏严重
- 建议先不搞，看实际使用效果再决定

### 5.4 多群监控

```json
// config.json
{
  "groups": [
    {"group_id": 123456789, "group_name": "击球区交流群", "forward_media": true},
    {"group_id": 987654321, "group_name": "期货实战群",     "forward_media": false}
  ]
}
```

每个群独立追踪，互不影响。飞书端可以发到不同群（通过 `feishu_chat_id` 配置）。

### 5.5 多个 QQ 账号

如果用户有多个 QQ 号在不同群：

**方案 A**：多个 NapCat 实例（每个实例一个 QQ 号）
- Docker Compose 里配多个 napcat service
- 每个绑定不同端口（3001, 3002, ...）
- 转发脚本连接所有 WebSocket

**方案 B**：一个 QQ 号加所有目标群
- 最简单，用一个小号加入所有需要监控的群
- 缺点：需要群主/管理员同意拉入

## 六、部署方案

### 6.1 服务器环境

| 项目 | 详情 |
|------|------|
| 服务器 | 阿里云轻量 116.62.173.189（与 ZSXQ Monitor 同一台） |
| 操作系统 | Alibaba Cloud Linux 3 |
| Docker | 需安装（目前未装） |
| Python | 3.6.8（系统自带） |
| 转发方式 | Docker Compose（NapCat + Bridge 在同一网络） |

### 6.2 Docker Compose 部署

```yaml
# docker-compose.yml
services:
  napcat:
    image: mlikiowa/napcat-docker:latest
    environment:
      - ACCOUNT=<QQ小号>
    ports:
      - "6099:6099"
      - "3001:3001"
    volumes:
      - ./napcat_data:/app/.config/QQ
    restart: unless-stopped

  bridge:
    build: ./bridge
    environment:
      - NAPCAT_WS=ws://napcat:3001
      - FEISHU_APP_ID=xxx
      - FEISHU_APP_SECRET=xxx
      - FEISHU_CHAT_ID=oc_xxx
    volumes:
      - ./data:/app/data          # SQLite DB 持久化
      - ./config.json:/app/config.json
    restart: unless-stopped
    depends_on:
      - napcat
```

### 6.3 Docker 安装（服务器上执行）

```bash
# Alibaba Cloud Linux 3 (兼容 CentOS 8)
yum install -y yum-utils
yum-config-manager --add-repo https://download.docker.com/linux/centos/docker-ce.repo
yum install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin
systemctl enable docker && systemctl start docker
docker --version
```

## 八、风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| QQ 小号被封 | 中 | 高 | 专用号、减少登录地切换、申诉解封 |
| NapCat 失效（QQ 升级） | 中 | 高 | 关注 NapCat 更新、保留 Claude 方案做备份 |
| NapCat Docker 镜像停更 | 低 | 中 | 有 Dockerfile 可自建 |
| 服务器 IP 被 QQ 风控 | 中 | 中 | 选国内节点、先在本地登录再迁移 |
| 飞书 API 频率限制 | 低 | 低 | 消息量不大（一个群一天几十条） |
| QQ 登录过期需重新扫码 | 中 | 中 | WebUI 远程扫码、告警通知 |
| Docker 与现有服务冲突 | 低 | 低 | 独立 Docker 网络，不冲突 |
